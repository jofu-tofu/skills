# Contract Testing

**Source:** Martin Fowler, "ContractTest" (2011) / Pact framework / Ian Robinson, consumer-driven contracts

**Impact: High** — Eliminates the dilemma between slow full-integration tests and mocks that lie. Essential for microservices and distributed systems.

---

## The Problem

When Service A calls Service B, you have three testing options:

| Option | Speed | Confidence | Problem |
|--------|-------|-----------|---------|
| Full integration (both services running) | Slow | High | Brittle, expensive CI, hard to parallelize |
| Unit test with mocks | Fast | Low | Mocks drift from reality; false confidence |
| **Contract tests** | **Fast** | **High** | **The right tool** |

---

## What Is a Contract?

A contract is a formal agreement between a **consumer** (the caller) and a **provider** (the service being called) about:
- What requests the consumer makes
- What responses the provider returns
- Under what conditions

Contracts are stored as files (typically JSON) and verified independently by each side.

---

## The Core Principle

**Consumer-driven contracts:** The consumer defines what they need. The provider verifies they honor it.

This is backwards from traditional API design (provider defines spec, consumers adapt). Consumer-driven contracts are better because they capture *actual usage* — not theoretical API capabilities.

---

## Why Contracts Beat Mocks

```pseudocode
// Mock-based: Mock is a snapshot of old API
stripe_mock = mock()
stripe_mock.charges.create.returns({charge_id: "ch_123"})
// Stripe renames field to {id: "ch_123"} — mock still passes

// Contract-based: Contract is verified against real Stripe
// When contract is run against new Stripe API:
// → Test fails because "charge_id" field no longer exists
// → You find out before deploying to production
```

---

## The Test Question

*"If the provider changes their API, will my tests catch it before I deploy?"*

With mocks: No. With contracts: Yes.

---

## Contract Testing vs. Other Test Types

| What you're verifying | Use |
|----------------------|-----|
| Business logic inside one service | Unit test |
| Two services communicate correctly | **Contract test** |
| Full user journey end-to-end | E2E test (few) |
| API schema validity | Schema validation (OpenAPI/JSON Schema) |

---

## Implementation Approaches

### Pact (most common)

Pact is the de facto standard for consumer-driven contract testing. It has implementations for most languages and a broker for storing/sharing contracts.

Flow:
1. Consumer writes test → generates a Pact file (the contract)
2. Pact file uploaded to Pact Broker
3. Provider downloads Pact file and runs verification
4. Both sides can deploy independently once contracts verified

### Spring Cloud Contract (Java ecosystem)

Provider-driven: provider defines contracts (as Groovy DSL or YAML), generates tests and stubs automatically. Consumers use the generated stubs.

### WireMock / API Mocking with Contracts

Record real API interactions → replay in tests → use recordings as contracts.

## Where Contract Tests Fit in the Pyramid

```
         ▲
        /E2E\ (very few — prove system works together)
       /─────\
      / Contract \ (moderate — prove services agree on interface)
     /────────────\
    / Integration  \ (some — prove individual service integrations work)
   /───────────────\
  /   Unit Tests    \ (many — prove individual logic works)
 /─────────────────\
```

Contract tests sit between unit and integration. They're faster than integration (no real services needed) but more reliable than mocks (contracts are verified against real providers).

---

## Anti-Patterns to Avoid

- **Testing business logic in contracts** — Contracts test interface agreement, not behavior inside the service
- **One-sided contracts** — Consumer writes contract but provider never verifies it
- **Manual contract maintenance** — Contracts should be generated by tests, not written by hand
- **Trusting unverified contracts** — A contract only has value if the provider has verified it recently
